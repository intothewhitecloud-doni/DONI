# Git 커밋 메시지 컨벤션 (Conventional Commits)

이 프로젝트는 커밋 메시지에 [Conventional Commits](https://www.conventionalcommits.org/ko/) 스타일을 따릅니다.
커밋 메시지를 일관되게 작성함으로써, 변경 이력을 명확하게 파악하고 자동 changelog 생성 등의 자동화를 가능하게 합니다.

---

## ✅ 커밋 메시지 형식

```
<type>(optional scope): <subject>

[optional body]

[optional footer]
```

- `type`: 변경의 종류 (아래 참고)
- `scope`: 선택사항, 변경된 모듈/파일/기능 범위
- `subject`: 간결한 설명 (50자 이내, 마침표 ❌)
- `body`: 선택사항, 변경 이유와 상세 내용
- `footer`: 선택사항, 이슈 연결 (`Closes #123` 등)

---

## ✍️ 작성 언어

- 커밋 메시지는 한글로 작성합니다.
- `type`, `scope`, trailer key, 이슈 번호, 버전 태그, 패키지명, API명, 코드 식별자처럼 기계 판독 또는 고유 명칭으로 필요한 값은 원문 표기를 허용합니다.
- trailer를 사용할 때도 값은 한글로 작성합니다.

---

## 🧾 문장 스타일

- `subject`는 개조식 명사형으로 작성합니다.
- 문장형 종결 표현인 `~한다`, `~했다`, `~합니다`, `~했습니다`는 피합니다.
- 여러 변경 이유나 구현 내용을 적을 때는 `body`를 `-` 목록으로 작성합니다.
- `body`는 변경 내용 나열보다 의도, 제약, 선택 이유를 우선합니다.
- `subject` 끝에는 마침표를 쓰지 않습니다.

예시:

```
feat(vault): PDF 미리보기 지원

- PDF 파일을 브라우저 내장 뷰어로 표시
- 같은 이름의 이미지와 PDF를 별도 파일로 보관

Constraint: 새 PDF 렌더링 의존성 추가 없이 구현 필요
Tested: npm run test:policy
```

---

## 📌 type 종류

| 타입       | 설명                                            |
| ---------- | ----------------------------------------------- |
| `feat`     | 새로운 기능 추가                                |
| `fix`      | 버그 수정                                       |
| `refactor` | 기능 변화 없이 내부 코드 개선                   |
| `docs`     | 문서 관련 변경 (README 등)                      |
| `style`    | 코드 포맷팅, 세미콜론 등 스타일 변경            |
| `test`     | 테스트 코드 추가/수정                           |
| `chore`    | 빌드, 설정, CI, 의존성 등 변경                  |
| `perf`     | 성능 개선                                       |
| `build`    | 빌드 시스템 또는 외부 의존성에 영향을 주는 변경 |
| `ci`       | CI 관련 설정 변경                               |
